01 · how the algorithm works

The application uses a method called K-Nearest Neighbours with Logistic Ridge Regression (KNN-LRR) to estimate the probability that a crime incident results in an arrest. This section explains what that means in plain terms.

step 1 · find similar past incidents

When you submit a query — specifying a crime type, a location on the map, a date, and a time — the algorithm searches the historical Chicago crime dataset (2002–2025) for the k incidents most similar to yours. Similarity is measured in terms of both where and when the crime occurred. Incidents that happened nearby and at a similar time of day, day of week, and time of year are considered closer neighbours.

step 2 · learn from those neighbours

Once the k nearest neighbours are identified, a logistic regression model is fitted locally on just those k records. Logistic regression is a statistical method that models the relationship between a set of input features and a binary outcome — in this case, whether an arrest was made (True or False). A Ridge penalty is applied to prevent the model from over-fitting to the small local sample.

step 3 · predict arrest probability

The locally-fitted model is then used to predict the probability of arrest for your specific query point. The output is a number between 0 and 1: values close to 1 indicate a high likelihood of arrest; values close to 0 indicate a low likelihood.

why local fitting?

Arrest patterns are not uniform across Chicago. They vary by neighbourhood, time of day, and crime type. By fitting the regression only on nearby incidents, the algorithm adapts to the local context of each query rather than applying a single global model to the entire city.

from-scratch vs. sklearn

The app offers two implementations of the same algorithm: a from-scratch version built without machine learning libraries, and a sklearn mirror that uses standard Python libraries as a validation benchmark. Both should produce predictions within 5 percentage points of each other for the same inputs.

02 · app inputs

The six inputs exposed in the “Will I be arrested?” dashboard. Each maps to one or more model features described in section 04.

inputui typedescription
crime type dropdown The category of crime being queried. Filters the dataset to records of that type before the neighbour search is run. Must match one of the primary_type values in the dataset (e.g. THEFT, BATTERY, NARCOTICS). The model is fitted only on the filtered subset, so predictions are specific to that crime type. Crime types only include those with more than 5000 cases.
location (click map) map pin The latitude and longitude of the query point, selected by clicking anywhere on the Chicago map. Represents where the crime occurred. These coordinates are two of the ten features used in distance computation. Only valid Chicago coordinates are accepted (~41.6–42.0°N, ~−87.9 to −87.5°W).
date date picker The calendar date of the query incident. Used to extract month, day_of_week, and day_of_year, which are then cyclically encoded as (sin, cos) pairs.
hour of day dropdown The hour at which the crime occurred (24-hour clock, e.g. 12:00 = noon). Cyclically encoded as (sin, cos) so that 23:00 and 00:00 are treated as close in time rather than far apart. One of the eight temporal features fed into the distance metric.
algorithm dropdown Which implementation to use: KNN (from scratch) or KNN (sklearn) — the validation mirror built with standard Python libraries. Both use identical feature sets, standardisation, and regularisation. Predictions should agree within |Δp| < 0.05 for the same inputs.
k slider The number of nearest neighbours to retrieve from the filtered dataset. Controls the size of the local neighbourhood used to fit the logistic regression. Smaller k = more sensitive to local patterns but higher variance; larger k = smoother estimates but less local. The app default is 25. Must be smaller than the number of records available for the selected crime type.
{# ── 3. Model output ─────────────────────────────────────── #}

03 · model output

The single value returned by the algorithm for each query.

outputrangedescription
arrest probability 0.0 – 1.0 The estimated probability that a crime incident with the given characteristics (crime type, location, date, and time) results in an arrest. Derived from the Boolean arrest variable in the dataset (True if an arrest was made, False otherwise). The model predicts this probability locally: a logistic regression is fitted on the k nearest neighbours of the query point, and the resulting model is used to predict the probability for the query itself. Across the crime types (with more than 5000 cases) in the dataset, approximately 28% of incidents result in an arrest — but rates vary substantially by crime type (e.g. narcotics arrests are disproportionately high; theft and criminal damage are low). The locally-fitted model reflects these differences by conditioning on crime type and spatiotemporal context.

08 · data notes & known issues

year 2001 excluded
A disproportionately high share of 2001 records are missing latitude/longitude, making spatial features unreliable for that year. All modelling uses 2002 onward.
collapse to incident level
The raw source data is person-level (multiple rows per case_number for multi-person incidents). We collapsed the data to one row per incident. For arrest, the max is taken (True if any person was arrested).
local parquet vs. pipeline output
The locally cached parquet (data/cleaned/chicago_crimes_cleaned.parquet) contains source columns only. Engineered columns (temporal + cyclic) are generated at runtime by preprocess_data() and are not stored in the parquet.
cyclic encoding rationale
Raw hour/month/day values cannot be used directly in Euclidean distance because the scale wraps around (e.g. hour 23 and hour 0 are 1 unit apart, not 23). Encoding each unit as (sin, cos) preserves this circularity in Euclidean space.
missing coordinates
~1% of records lack latitude/longitude. These are dropped before spatial modelling (NearestNeighbors and haversine distance require valid coordinates).
